## 🎯 TL;DR（一句话总结）

FunSearch 首次将预训练 LLM 与系统性评估器配对，通过进化式程序搜索在函数空间中发现可验证的新知识——在 cap set 问题上打破 20 年未动的渐近下界，在在线装箱问题上发现超越传统启发式的新算法。

---

## ❓ 核心问题

LLM 在科学发现中面临一个根本性矛盾：**它既能生成看似合理的创造性想法，又频繁产生幻觉（confabulations）**。在数学和算法领域，一个" plausible but incorrect "的命题毫无价值。因此核心问题是：

**如何利用 LLM 的创造性生成能力，同时通过可验证的反馈机制确保发现的真实性，从而使其能够解决已建立的开放科学问题？**

FunSearch 的回答是：不直接让 LLM 输出答案，而是让 LLM 在**函数空间**中搜索**生成解的程序**——评估器作为不可博弈的选择压力，淘汰错误程序，保留正确且优秀的程序，形成进化闭环。

## 🔧 方法

### 核心架构：LLM + Evaluator 的进化闭环

```
┌─────────────────────────────────────────┐
│  问题规格（Specification）               │
│  evaluate函数 + solve骨架 + 初始程序    │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  程序池（Programs Database）             │
│  多岛屿模型（Multi-Island）               │
│  保留多样性，避免局部最优                  │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  Prompt构建（Best-Shot Prompting）        │
│  从程序池采样k=2个高分程序              │
│  按得分排序，附加版本后缀（v0, v1...）   │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  LLM（Codey / PaLM 2 代码版）            │
│  预训练模型，无需微调                     │
│  基于prompt生成新程序变体                 │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  评估器（Evaluator）                     │
│  隔离执行代码，计算客观得分                │
│  错误程序（超时/内存/无效输出）丢弃        │
└─────────────────┬───────────────────────┘
                  │
         ┌────────┴────────┐
         ▼                 ▼
    ┌─────────┐      ┌─────────────┐
    │  通过   │      │   未通过    │
    │ 加入    │      │   丢弃      │
    │ 程序池  │      │             │
    └────┬────┘      └─────────────┘
         │
         └──────→ 反馈至 LLM，迭代进化
```

### 关键设计要素

#### 1. 程序骨架（Skeleton）与优先级函数（Priority Function）
- **核心洞察**：不进化整个程序，而是将程序分解为**骨架**（已知结构的 boilerplate 代码）和**待进化部分**（关键逻辑）。
- **Cap set 问题**：骨架是一个贪心算法，每次选择最高优先级的向量加入 cap set。FunSearch 进化的对象是 `priority: Z₃ⁿ → R` 函数。
- **Bin packing 问题**：骨架是在线决策框架，进化的是 `heuristic` 函数，决定每个物品放入哪个箱子。
- **优势**：聚焦 LLM 的计算资源于最难的部分，避免重复生成已知结构（减少错误概率）。

#### 2. 最佳样本提示（Best-Shot Prompting）
- 从程序池中采样 k=2 个程序（按得分加权），按得分排序后附加版本后缀（`priority_v0`, `priority_v1`）。
- 在 prompt 末尾添加待生成函数的头部（`priority_v2`）。
- **为什么 k=2？** 实验表明两个程序比一个好，超过两个收益递减。多个程序使 LLM 能发现跨程序的模式并泛化。

#### 3. 多岛屿进化（Multi-Island Evolutionary Method）
- 程序池被划分为 m 个独立的**岛屿**（子种群），各自独立进化。
- **信息流动机制**：每 4 小时淘汰得分最低的 m/2 个岛屿，用存活岛屿中最佳程序的克隆重新初始化。
- **岛屿内聚类**：按程序的**签名**（signature，即各输入上的得分元组）聚类。采样时先采样聚类（高分偏好），再采样程序（短程序偏好）。
- **目的**：维持种群多样性，避免所有实验陷入同一局部最优。

#### 4. 分布式异步架构
- **三种 worker**：Programs Database（存储/服务程序）、Samplers（LLM 生成新函数）、Evaluators（评估程序）。
- **并行策略**：15 个 sampler + 150 个 CPU evaluator（5 台 CPU 服务器，每台 32 个并行评估）。
- **成本优化**：评估器在廉价 CPU 上运行，sampler 在带加速器的机器上运行，保持整体成本低廉。

## 📊 关键实验与结果

### 实验一：Cap Set 问题（极值组合学）

**问题定义**：在 Z₃ⁿ 中寻找最大的 cap set——即没有三个向量之和为零的集合。被陶哲轩称为"perhaps my favourite open question"。

**FunSearch 成果**：

| 维度 n | 之前已知最大 cap set | FunSearch 发现 | 突破 |
|--------|---------------------|---------------|------|
| n = 8 | 496 | **512** | ✓ 新纪录 |
| n = 9 | — | 扩展至更大维度 | — |

- **n = 8 维的 512-cap**：FunSearch 从零发现，无需被显式教授任何低维 cap set 的组合方法。之前最好的方法依赖复杂的多 cap set 组合技巧（Edel & Bierbrauer 2001, 2004）。
- **可解释性优势**：发现的不仅是一个 512 个向量的集合，而是**生成它的程序**。通过人工简化该程序，研究者获得了对构造本质的理解（与 Hill cap 的某些性质相似）。
- **统计显著性**：n=8 的 512-cap 发现并不容易——140 次独立实验中仅 4 次成功。

**渐近下界突破（Admissible Sets 框架）**：
- Cap set 的容量 C = supₙ cₙ^(1/n) 是加性组合学的核心问题。
- 2017 年 Ellenberg & Gijswijt 证明上界 C ≤ 2.756。
- **FunSearch 将下界提升至 20 年来最大改进**，通过发现新的 admissible set 构造。
- 所有独立实验均超越之前最佳下界；60% 的 I(12,7) 实验找到完整大小的 admissible set。

### 实验二：在线装箱问题（Online Bin Packing）

**问题定义**：物品依次到达，必须立即决定放入哪个箱子（不可重排），目标是最小化总箱子数。

**FunSearch 成果**：
- 发现新的启发式算法，在广泛研究的基准分布上**超越传统启发式**（First Fit, Best Fit, Harmonic 等）。
- 发现的启发式具有**泛化能力**：在训练分布上进化，但在更大实例上同样有效。
- **实际应用潜力**：在线装箱是作业调度（job scheduling）的核心抽象，新启发式可直接应用于数据中心调度优化。

## 🔍 我的批判

### 优点

1. **"搜索程序而非搜索解"的范式突破**：传统计算机搜索输出解本身（如 cap set 的向量列表），FunSearch 输出的是**生成解的程序**。这带来三大优势：可解释性（人类可阅读代码理解构造本质）、可扩展性（程序可生成任意规模的解）、可部署性（标准编程语言的决策程序比神经网络更易部署和验证）。

2. **LLM 作为"语义感知变异算子"的进化范式**：传统遗传编程的变异是随机的符号操作（删除/交换代码行），需要人工设计原语集且问题依赖。FunSearch 中 LLM 利用其在海量代码上预训练获得的知识，进行**有意义的、上下文感知的代码重组**——这是进化计算的新范式，论文称之为"mitigates several issues in traditional genetic programming"。

3. **评估器作为不可博弈的选择压力**：评估器的存在使得进化系统有一个**客观的、不可被 LLM 欺骗的反馈源**。这与纯 LLM 自举（self-improvement via LLM-only feedback）有本质区别——后者存在奖励黑客（reward hacking）风险，因为 LLM 可能学会生成"看起来好"但实质上错误的答案。

4. **多岛屿模型的设计智慧**：不仅是简单的并行化，而是通过周期性淘汰低分岛屿实现"自然选择"级别的实验筛选。这相当于同时运行多个独立实验，自动杀死陷入局部最优的实验，从最有希望的实验重新启动。

### 局限

1. **评估函数的刚性约束**：FunSearch 的成功严格依赖于"evaluate 函数可高效计算"的前提。许多开放科学问题（如数学猜想证明、理论物理模型构建）缺乏这样的评估函数——你无法自动验证一个证明是否正确（除非形式化），也无法自动评分一个物理理论的优美程度。这使得 FunSearch 的适用范围被限制在**组合优化和算法设计**领域。

2. **骨架设计仍依赖人类专家**：论文声称"without problem-specific tuning"，但实际上骨架（skeleton）的设计质量直接决定搜索空间的质量。cap set 问题的贪心骨架、bin packing 的在线决策骨架——这些都需要领域专家的先验知识。FunSearch 自动化了" hardest part "的进化，但**框架本身的设定仍是人工的**。

3. **LLM 的"创造性天花板"**：FunSearch 使用预训练 LLM（Codey/PaLM 2），其创造性本质上受限于训练数据中的代码模式。论文承认"results are not too sensitive to the exact choice of LLM"——这既是优点（鲁棒性）也是警示：如果 LLM 的训练语料中完全没有某种数学结构的痕迹，FunSearch 可能永远无法发现它。真正的突破性数学发现（如椭圆曲线与模形式的联系）往往需要训练数据之外的"跳跃"。

4. **可复现性与计算成本**：论文报告使用约 10⁶ 次 LLM 采样。虽然作者强调"overall cost low"，但对于大多数学术实验室而言，15 sampler + 150 evaluator 的分布式设置仍非 trivial。更重要的是，140 次实验中仅 4 次发现 512-cap——这意味着**成功具有显著的随机性**，小规模复现可能完全失败。

### 疑问

- **与形式化证明的衔接**：论文提到"automatic theorem proving still lags behind the frontier of human knowledge"。FunSearch 能否与形式化证明系统（如 Lean/Coq）结合，使发现的程序不仅通过数值评估验证，还能获得形式化正确性证明？
- **神经-符号混合的可能性**：纯代码表示的启发式缺乏梯度优化的精细调节。对于高频决策问题（如实时调度），是否可以将 FunSearch 发现的启发式编译为可微分的神经策略，实现符号可解释性与神经优化效率的兼得？
- **开放世界中的评估器设计**：FunSearch 的评估器是确定性的、客观的。但在更开放的领域（如科学假设生成、实验设计），评估器本身可能是不完美的代理（proxy）。当评估器与真实目标不一致时，进化系统会如何表现？

### 与 Voyager 的 Skill Library 机制对比

| 维度 | FunSearch (Programs Database) | Voyager (Skill Library) |
|------|-------------------------------|------------------------|
| **进化驱动** | 评估器驱动的**外部选择压力** | 任务完成驱动的**内在动机** |
| **存储单元** | 函数/程序片段（数学/算法逻辑） | 可执行代码技能（Minecraft API 调用） |
| **检索机制** | 多岛屿 + 签名聚类 + 得分加权采样 | 向量数据库语义检索（embedding） |
| **多样性维持** | 显式多岛屿模型 + 周期性淘汰 | 隐式依赖向量空间的语义覆盖 |
| **评估标准** | 客观数值得分（cap set 大小、bin packing 效率） | 任务完成与否（二元成功/失败） |
| **LLM 角色** | **变异算子**——基于优秀程序上下文生成变体 | **技能生成器+课程设计者**——生成新技能代码和探索目标 |
| **人类可解释性** | 高——发现的程序可被数学家阅读和理解 | 中——技能代码可阅读，但组合逻辑复杂 |
| **闭环类型** | **评估→选择→繁殖**的达尔文式闭环 | **执行→反思→存储**的经验积累式闭环 |

**关键差异**：
1. **选择压力来源不同**：FunSearch 的评估器是**客观的、预先定义的数学函数**，不存在博弈空间；Voyager 的任务完成评估是**环境反馈**，可能受 Minecraft 物理引擎的噪声和偶然性影响。
2. **进化方向不同**：FunSearch 是**定向优化**（最大化 cap set 大小），Voyager 是**开放探索**（"尽可能发现更多事物"）。前者收敛，后者发散。
3. **程序复用机制不同**：FunSearch 通过 best-shot prompting 将优秀程序直接注入 LLM 上下文实现"继承"；Voyager 通过向量检索实现技能的语义复用。前者是**显式代码级遗传**，后者是**隐式语义级检索**。
4. **可扩展性判断**：FunSearch 的分布式架构（15 sampler + 150 evaluator）更适合**计算密集型、评估昂贵的领域**；Voyager 的架构更适合**交互密集型、反馈即时的环境**。

### 启发

1. **创造力 = 约束 + 变异 + 选择**：FunSearch 的惊人发现不是来自无约束的开放生成，而是来自**严格约束空间**（骨架 + 评估器）中的定向进化。这提示：自进化系统的设计应优先定义**不可博弈的评估函数**，再让 LLM 在约束下自由变异。

2. **程序是最优的进化表示**：相比参数向量或自然语言策略，代码具有**可执行性、可组合性、可验证性**三重优势。FunSearch 和 Voyager 不约而同选择代码作为进化单元，这不是偶然——代码是连接"生成"与"现实"的最短桥梁。

3. **LLM 不应替代进化，而应嵌入进化**：FunSearch 没有抛弃遗传算法，而是将 LLM **嵌入**到变异步骤中。这是关键架构决策：LLM 增强特定环节（创造性突变），而非取代整个进化流程。自进化 Agent 的设计应遵循"LLM 嵌入经典框架"而非"LLM 取代经典框架"的原则。

4. **可验证性是科学发现的护城河**：FunSearch 仅在可验证领域成功，这强调了 [[Verifiable Evolution]] 原则。自进化 Agent 应优先在具有客观反馈环的能力上启动进化，避免在主观评价领域盲目进化。
