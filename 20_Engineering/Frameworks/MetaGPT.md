## 🏗️ 架构概览

MetaGPT 的核心创新在于**将人类软件工程团队的协作模式编码为 Multi-Agent 系统**。它通过角色分工和标准化流程（SOP）实现复杂任务的分解与协作。

```
┌─────────────────────────────────────────┐
│              用户需求/产品概念            │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  产品经理 (Product Manager)               │
│  输出：产品需求文档 (PRD)                 │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  架构师 (Architect)                     │
│  输出：系统设计文档、技术选型             │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  项目管理者 (Project Manager)             │
│  输出：任务分解、进度规划               │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  工程师 (Engineer)                      │
│  输出：代码实现                          │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  QA (Quality Assurance)                  │
│  输出：测试用例、代码审查、Bug 报告       │
└─────────────────────────────────────────┘
                  │
                  └──────→ 反馈迭代至 PM/工程师
```

所有角色通过**共享消息总线**通信，每个角色具备独立的上下文记忆和职责边界。

---

## 🧩 核心抽象

| 组件 | 职责 | 输入 | 输出 |
|------|------|------|------|
| **角色 (Role)** | 封装特定职能的 Agent 行为模式 | 消息总线中的相关消息 | 结构化文档/代码/决策 |
| **SOP 标准化流程** | 定义角色间的协作协议与交付物标准 | 角色当前状态 | 下一步行动约束 |
| **共享消息总线** | 全局发布-订阅通信机制 | 各角色的产出 | 广播至订阅者 |
| **环境 (Environment)** | 维护共享状态与资源（代码库、文档库） | 角色操作 | 更新后的共享资源 |

## ⚖️ 优劣分析

### 优点
- **结构化协作**：SOP 将模糊的自然语言协作转化为可预期的流程，显著降低多 Agent 系统的混沌度
- **角色专业化**：每个 Agent 只需专注单一职能，减少上下文切换，提升单环节质量
- **可解释性强**：标准化交付物（PRD、设计文档）使系统行为透明可追溯
- **人类对齐**：直接映射人类团队组织方式，降低理解和调试门槛

### 局限
- **灵活性不足**：SOP 是硬编码的，面对非软件工程类任务（如科学研究、创意写作）适配困难
- **角色僵化**：固定角色数量难以扩展，新增角色需要重新定义整个协作拓扑
- **通信开销**：消息总线的广播机制在角色增多时产生冗余信息，可能淹没关键信号
- **反馈循环浅层**：QA→工程师→PM 的反馈链是线性的，缺乏跨角色的深度共识机制

## 🔄 与自进化的关系

MetaGPT 是自进化 Agent 系统的**组织范式参考**，其多角色结构暗示了进化的分布式本质：

1. **角色 = 进化单元**：每个角色可视为独立进化的子系统。自进化可发生在角色内部（如工程师改进编码策略），也可发生在角色间（如 PM 学习更优的需求分解模式）。

2. **SOP 作为可进化对象**：当前 SOP 是静态的，但可扩展为 **[[Adaptive SOP]]**——让系统根据任务类型和历史表现动态调整协作流程。这是 MetaGPT 向自进化升级的关键路径。

3. **消息总线 → 进化信息通道**：共享通信层不仅是协作基础设施，也是**进化信号的载体**。角色间的反馈（如 QA 发现的 Bug 模式）可提炼为全局启发式，指导所有角色的行为调整。

4. **与 [[Multi-Agent Co-evolution]] 的映射**：MetaGPT 的固定角色分工是"前协调"（ex-ante coordination），而自进化需要"后适应"（ex-post adaptation）——角色职责本身应随任务演化而重组。

## 💡 给我的启发

1. **组织即算法**：MetaGPT 证明了一个深刻命题——**协作结构本身就是问题求解策略的一部分**。自进化不应只关注单个 Agent 的能力提升，更应关注 Agent 间**组织形态的进化**。

2. **SOP 的双刃剑**：标准化流程带来稳定，但也锁定创新。启发是：自进化系统需要 **"元-SOP"**——一套管理 SOP 自身变更的规则，允许系统在稳定期遵守流程，在瓶颈期打破流程。

3. **角色的动态重组**：固定角色是认知劳动分工的初级形态。更高级的形态是 **[[Emergent Role]]**——Agent 根据任务需求自发形成临时角色，任务完成后角色解散。这比预定义角色更接近生物系统的自组织。

4. **文档作为中间表示**：MetaGPT 要求角色间通过结构化文档通信，这实际上是 **[[Shared Mental Model]]** 的工程实现。自进化系统中，共享表示的质量直接决定协作上限——值得投入进化资源优化表示本身。
