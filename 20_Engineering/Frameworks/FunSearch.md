## 🏗️ 架构概览

FunSearch（Function Space Search）是 Google DeepMind 于 2023 年底发布的进化式代码发现框架。其核心创新在于**将 LLM 的代码生成能力与遗传算法的进化机制结合**，在函数空间中搜索解决数学和算法问题的新方案。

```
┌─────────────────────────────────────────┐
│           问题定义（代码框架）             │
│  评估函数 + 种子程序 + 程序骨架             │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  程序池（Programs Database）             │
│  维护历史上所有被验证过的程序变体           │
│  按得分排序，保留精英个体                   │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  LLM（Codey / PaLM 2 代码版）            │
│  从程序池采样优秀程序作为上下文             │
│  "创造性地构建"新程序变体                   │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  自动评估器（Evaluator）                 │
│  执行生成的代码，计算客观得分                │
│  判断：优于已知最优解？                     │
└─────────────────┬───────────────────────┘
                  │
         ┌────────┴────────┐
         ▼                 ▼
    ┌─────────┐      ┌─────────────┐
    │  通过   │      │   未通过    │
    │ 加入    │      │   丢弃      │
    │ 程序池  │      │             │
    └────┬────┘      └─────────────┘
         │
         └──────→ 反馈至 LLM，迭代进化
```

关键洞察：**LLM 负责"创造性突变"，评估器负责"自然选择"**，两者形成闭环进化系统。这是首次 LLM 在可验证领域做出全新科学发现（cap set 问题、bin packing 问题）。

## 🧩 核心抽象

| 组件 | 职责 | 输入 | 输出 |
|------|------|------|------|
| **程序池 (Programs Database)** | 存储并管理所有被验证的程序变体，实现精英保留 | 评估通过的新程序 | 按得分排序的程序集合 |
| **LLM 突变器 (LLM Mutator)** | 基于优秀程序上下文，生成创造性代码变体 | 采样的精英程序 + 提示模板 | 新程序代码 |
| **评估器 (Evaluator)** | 隔离执行代码，客观计算问题得分 | 生成的程序 + 测试用例 | 标量得分 / 执行错误 |
| **进化循环 (Evolutionary Loop)** | 协调采样→突变→评估→选择的迭代流程 | 当前程序池状态 | 更新后的程序池 |
| **问题框架 (Problem Frame)** | 定义程序接口、评估函数和初始种子 | 领域专家设计 | 可执行的代码骨架 |

## ⚖️ 优劣分析

### 优点
- **可验证发现**：评估器提供客观反馈，避免了 LLM 幻觉问题，确保发现的正确性
- **创造性突破**：在 cap set 问题等数学难题上发现了超越人类已知最优解的新构造
- **通用框架**：不依赖特定领域知识，只要问题可编码为"程序 + 评估函数"即可应用
- **可解释产出**：发现以可执行代码形式呈现，人类可阅读、验证和改进
- **进化规模效应**：程序池随迭代积累优质变体，LLM 的上下文学习能力使其能"站在巨人肩膀上"

### 局限
- **评估瓶颈**：需要精确、高效、可自动化的评估函数，许多开放性问题难以如此编码
- **领域依赖**：问题框架的设计质量直接决定搜索空间质量，仍需人类专家参与
- **计算成本**：每次迭代需调用 LLM + 执行评估，大规模搜索成本高昂
- **代码安全性**：自动生成的代码需要沙箱隔离执行，防止恶意或破坏性操作
- **继承者出现**：2025 年 DeepMind 发布 AlphaEvolve，作为 FunSearch 的精神续作，在代码进化、评估、上下文和搜索方面大幅改进

## 🔄 与自进化的关系

FunSearch 是自进化 Agent 系统的**进化引擎原型**——它展示了 LLM 如何作为"智能突变算子"参与进化：

1. **LLM 作为进化算子**：传统遗传算法的突变是随机的，而 FunSearch 中 LLM 的突变是**语义感知的**——它理解代码结构，能进行有意义的重组和创新。这代表了进化计算的新范式：**引导式进化（Guided Evolution）**。

2. **评估即现实**：评估器的客观反馈是进化系统的"环境选择压力"。FunSearch 证明，自进化系统必须有一个**不可博弈的反馈源**（如单元测试、数学验证），否则进化会偏离真实目标。

3. **程序池 = 进化记忆**：程序池不仅是存储，更是**种群多样性管理**。保留精英 + 淘汰劣质的机制对应自进化中的 **[[Selective Retention]]**。程序池的维护策略（如何平衡探索与利用）直接影响进化效率。

4. **与 [[Code-as-Genome]] 的映射**：FunSearch 将代码视为可进化的基因组，评估器作为适应度函数。这直接启发了自进化 Agent 的"代码自修改"能力——Agent 不仅能执行代码，还能进化自身代码。

## 💡 给我的启发

1. **创造力需要约束**：FunSearch 的惊人发现发生在**严格约束空间**（特定问题框架 + 客观评估）中，而非无约束的开放生成。这提示自进化系统：**进化不是越自由越好，而是在正确约束下的定向搜索**。

2. **LLM 是进化加速器，不是进化替代者**：LLM 没有取代遗传算法，而是**增强**了突变步骤。启发是：自进化系统应将 LLM 嵌入经典进化框架的特定环节（突变、交叉、选择），而非用 LLM 完全替代整个进化过程。

3. **可验证性是进化的前提**：FunSearch 仅在可验证领域成功，这强调了 **[[Verifiable Evolution]]** 原则。自进化 Agent 应优先在具有客观反馈环的能力上启动进化（如代码正确性、任务完成率），而非在主观评价领域盲目进化。

4. **代码是最通用的进化表示**：FunSearch 选择代码作为进化单元，因为代码是**可执行、可组合、可验证**的通用表示。自进化 Agent 的"基因组"也应优先考虑代码形式，而非参数向量或自然语言策略。

## 📡 最新状态（基于搜索验证）

- **发布时间**：2023 年 12 月（Nature 论文发表）
- **2024-2025 关键进展**：
  - **AlphaEvolve 发布（2025年5月）**：DeepMind 宣布 FunSearch 的精神续作 AlphaEvolve，作为"进化式编码 Agent"，由 Gemini 模型驱动，在代码进化、评估机制、上下文管理和搜索策略上全面升级
  - AlphaEvolve 已应用于 Google 内部工程挑战（如数据中心调度优化）和数学问题求解
  - FunSearch 的学术影响持续：2025 年仍有相关教程和扩展研究（如 SL Math Summer School 的 FunSearch 讲座）
  - 社区复现：多个开源实现出现，将 FunSearch 思想应用于更广泛的算法发现任务
- **状态评估**：FunSearch 作为原始框架已进入"经典参考"阶段，实际应用建议关注 AlphaEvolve
